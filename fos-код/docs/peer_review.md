# Регламент peer-review

**Дисциплина:** Методы оптимизации в системах искусственного интеллекта  
**Оценочное средство:** ОС-3 (10% итоговой оценки)

## 1. Назначение peer-review

Peer-review — формирующее оценочное средство, направленное на развитие:
- аналитических навыков (критический анализ методов оптимизации);
- коммуникативных компетенций (аргументированная обратная связь);
- инженерной культуры (оценка воспроизводимости кода).

## 2. Порядок проведения

1. Преподаватель формирует пары: каждая работа получает **2 независимых рецензии**; каждый студент рецензирует **2 работы**.
2. Рецензент получает доступ к репозиторию, инструкции запуска и демонстрационным материалам.
3. Срок заполнения шаблона — **5 календарных дней**.
4. Автор получает обезличенные рецензии; ответная реплика (rebuttal) приветствуется.
5. Преподаватель оценивает **качество самой рецензии** по 10-балльной шкале (см. раздел 6).

## 3. Метаданные рецензии

Заполняются в шаблоне `src/templates/peer_review_form.md`:
- Идентификатор рецензии
- Название работы автора
- Дата рецензирования
- Профильная ИИ-роль автора (ML Researcher / AI Security Engineer)
- Ссылка на репозиторий

## 4. Оценочный лист по критериям (10-балльная шкала)

По каждому критерию рецензент выставляет балл от 0 до 10 и оставляет содержательный комментарий (2–5 предложений). Комментарий **обязателен** для баллов ≤ 5 и ≥ 9.

| Код | Критерий | Пояснение |
|-----|----------|-----------|
| P1 | Ясность постановки задачи | Задача оптимизации понятна; целевая модель и данные зафиксированы |
| P2 | Обоснованность выбора методов оптимизации | Выбор SGD/Adam/L-BFGS/иных обоснован относительно задачи |
| P3 | Работа с данными | Источники, предобработка, качество данных описаны и проверяемы |
| P4 | Качество реализации оптимизаторов | Корректность кода; работа с градиентами; использование фреймворков |
| P5 | Настройка гиперпараметров | Применение grid/random/Bayesian search; использование Optuna |
| P6 | Эксперименты и метрики | Метрики релевантны; learning curves; выводы аргументированы |
| P7 | Анализ устойчивости и робастности | Тестирование на шуме, adversarial-примерах; оценка деградации |
| P8 | Репозиторная документация | README, PROJECT_BRIEF, инструкции; понятны стороннему инженеру |
| P9 | Соответствие индикаторам роли КРМ | Проект отвечает индикаторам выбранной ИИ-роли |
| P10 | Общая инженерная культура | Структура, читаемость кода, тесты, версии зависимостей, лицензия |

## 5. Ролевой блок рецензии (по КРМ)

Рецензент дополнительно оценивает, насколько работа отвечает индикаторам «профильной» ИИ-роли, используя портрет уровня 4 / 6 / 8 / 10 из `src/rubrics/role_portraits.md`.

| Профильная роль | Проверяемые индикаторы | Балл 4 / 6 / 8 / 10 |
|-----------------|------------------------|---------------------|
| ML Researcher | MF-3.1, MF-3.2, ML-3.1, ML-3.2, ML-3.3 | ___ |
| AI Security Engineer | ML-3.3, ML-5.1, ML-5.2, ML-5.3 | ___ |

## 6. Сильные и слабые стороны, рекомендации

Заполняются в открытом виде в шаблоне рецензии.

## 7. Итоговая рекомендация рецензента

- Итоговый балл проекта: ___ / 10
- Рекомендация к защите: ☐ рекомендовать / ☐ рекомендовать с доработкой / ☐ не рекомендовать

## 8. Оценивание качества самой рецензии (для преподавателя)

| Балл | Характеристика рецензии |
|------|-------------------------|
| 10 | Аргументированная, конструктивная, содержит проверяемые рекомендации, ролевой блок обоснован |
| 8 | Полная и содержательная, отражает большинство критериев |
| 6 | Формально полная, комментарии частично поверхностны |
| 4 | Минимально приемлемая: заполнены баллы, комментарии кратки |
| 0–3 | Формально пустая; рекомендации отсутствуют |
